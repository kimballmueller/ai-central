# Professional Client Proposal Skill

A reusable guide for drafting client-ready proposals that are clear, persuasive, and easy to sign.

## When to use

- Drafting a new proposal or SOW for a prospect
- Re-scoping an existing engagement
- Responding to an RFP or inbound inquiry
- Converting a discovery call into a written offer

## Tone

- **Confident, not boastful.** State capabilities as facts; skip superlatives ("world-class", "cutting-edge").
- **Plain English.** Short sentences. No jargon the client did not use first.
- **Client-centered.** Lead with their outcome, not our process. Use "you" more often than "we".
- **Specific over vague.** "Cut onboarding from 14 to 3 days" beats "improve onboarding efficiency".
- **Active voice.** "We will deliver X" not "X will be delivered".
- **Warm but professional.** Contractions are fine ("we'll", "you'll"). No slang, no emojis.

## Formatting rules

- **Length:** 2–5 pages typical. Never pad. If it can be cut without losing meaning, cut it.
- **Headings:** H1 for proposal title only. H2 for sections. H3 sparingly for sub-sections.
- **Lists:** Use bullets for 3+ parallel items. Use numbered lists only for ordered steps or phases.
- **Tables:** Use for pricing, timelines, and deliverable breakdowns. Avoid for prose.
- **Bold:** Reserve for key terms, deliverable names, and the price.
- **Dates:** Always absolute (e.g., "June 15, 2026"), never relative ("next month").
- **Currency:** Always include currency code on first mention ($12,000 USD).
- **Pronouns:** "We" = the firm. "You" = the client. Never switch mid-document.
- **Names:** Use the client's company name in the proposal title and at least once per major section.

## Section order

Every proposal follows this order. Do not reorder. Omit a section only if it genuinely does not apply, and note why in a comment.

### 1. Cover / Title block
- Proposal title (format: "Proposal for [Client Name]: [Project Name]")
- Prepared for: client contact name, title, company
- Prepared by: your name, company
- Date prepared
- Proposal validity (e.g., "Valid through July 15, 2026")

### 2. Executive Summary
2–4 sentences. What the client wants, what you'll deliver, the headline outcome, and the price range. A reader who only reads this section should understand the offer.

### 3. Background / Understanding
Reflect back what you heard in discovery. Demonstrates you listened. 1–2 short paragraphs. End with the core problem in one sentence.

### 4. Objectives
3–5 bullets. Each objective is a measurable outcome from the client's perspective, not an activity. "Reduce manual reporting time by 50%" not "Build a reporting dashboard".

### 5. Scope of Work
The core of the proposal. For each workstream or phase:
- **Name** of the workstream
- **What's included** (specific deliverables)
- **What's not included** (explicit exclusions to prevent scope creep)
- **Acceptance criteria** (how we'll know it's done)

### 6. Approach / Methodology
Brief. How you'll work, not what you'll deliver. Cover: discovery, build, review cadence, handoff. Skip if the client already knows your process.

### 7. Timeline
Table with phases, start dates, end dates, and key milestones. Use absolute dates. Note any client dependencies that affect timing.

### 8. Investment / Pricing
- Lead with the total. Break it down by phase or deliverable.
- Show payment schedule (e.g., 50% on signing, 50% on delivery).
- Include what's covered and what would be additional (travel, third-party tools, etc.).
- If offering tiers, present 3 options max (Good / Better / Best). Recommend one.

### 9. Team
Who will work on this and their role. One sentence per person. Skip headshots and long bios in the proposal — link to them.

### 10. Terms
- Validity window
- Payment terms (net 15, net 30)
- Cancellation / kill-fee policy
- IP ownership
- Confidentiality reference (or link to MSA if one exists)

### 11. Next Steps
3 short bullets. Make it frictionless: (1) sign, (2) kickoff date, (3) first deliverable. Include the signature block or a clear "Reply to accept" instruction.

### 12. Appendix (optional)
Case studies, detailed assumptions, technical specs. Only include if asked. Keep the main proposal lean.

## Quality checklist

Before sending, verify:

- [ ] Client name is spelled correctly in every instance
- [ ] All dates are absolute and consistent
- [ ] Total price appears in the executive summary and pricing section and matches
- [ ] Every objective maps to at least one deliverable in scope
- [ ] Every deliverable maps to at least one objective
- [ ] Exclusions are explicit, not implied
- [ ] No placeholder text ([CLIENT], TBD, lorem ipsum)
- [ ] Validity date is in the future
- [ ] Signature block or acceptance instruction is present
- [ ] Read aloud: any sentence that's hard to say is hard to read

## Anti-patterns

Avoid these:

- **Process theater.** Long methodology sections nobody reads.
- **Padding.** "We are excited to present this proposal..." Cut it.
- **Hedge words.** "We may", "we could potentially", "subject to". Commit or remove.
- **Mystery pricing.** "Contact us for pricing" — if you're sending a proposal, the price goes in it.
- **Open-ended scope.** "Ongoing support as needed" — define hours, response times, and renewal terms.
- **Buried CTA.** The client should never have to hunt for how to say yes.

## Output format

When generating a proposal, output it as a single markdown document ready to be exported to PDF or pasted into a docs tool. Use the section order above. Ask the user for any missing inputs (client name, scope, budget, dates) before drafting — do not invent them.
